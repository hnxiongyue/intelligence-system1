## 二、Dify 方案详解

### 2.1 Dify 是什么？

Dify 是一个开源的 LLM 应用开发平台，支持：
- 可视化工作流编排
- 内置多种 LLM（GPT、Claude、DeepSeek 等）
- 支持工具调用（类似 MCP）
- 支持知识库（向量数据库）
- 支持 API 调用和 Webhook

**官网**：https://dify.ai/

### 2.2 Dify 实现情报系统的方案

#### 架构设计

```
┌─────────────────────────────────────────────────────┐
│              定时触发（外部 Cron）                    │
│                      ↓                               │
│              调用 Dify Workflow API                  │
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│                 Dify 工作流                          │
├─────────────────────────────────────────────────────┤
│  1. HTTP 请求节点 → 调用 Firecrawl API 采集          │
│  2. 代码节点 → 数据清洗和去重                        │
│  3. LLM 节点 → AI 分析（影响评估、分类）              │
│  4. 条件分支 → 按优先级路由                          │
│  5. HTTP 请求节点 → 钉钉 Webhook 推送                │
│  6. 知识库节点 → 保存到向量数据库                    │
└─────────────────────────────────────────────────────┘
```

#### 优势

✅ **可视化编排**：拖拽式配置，无需写代码
✅ **内置 LLM**：直接配置 API Key，无需封装
✅ **内置知识库**：自动向量化，支持历史检索
✅ **支持工具调用**：可以调用外部 API
✅ **快速迭代**：修改工作流立即生效

#### 劣势

❌ **采集能力有限**：HTTP 请求节点功能简单，复杂网站难处理
❌ **代码节点限制**：Python 代码有沙箱限制，不能安装任意库
❌ **调试困难**：工作流调试不如纯代码方便
❌ **性能限制**：大量数据处理可能较慢

### 2.3 Dify 实施方案

#### 方案 A：纯 Dify（最简单）

**适用场景**：数据源简单，只需要基础采集

**实现步骤**：
1. 部署 Dify（Docker 或云端）
2. 创建工作流
3. 配置 HTTP 节点调用 Firecrawl
4. 配置 LLM 节点分析
5. 配置 Webhook 推送
6. 外部 Cron 定时触发

**优点**：最快，1-2 天搞定
**缺点**：采集能力弱，扩展性差

#### 方案 B：Dify + 自定义采集脚本（推荐⭐⭐⭐⭐⭐）

**适用场景**：需要复杂采集，但希望降低开发量

**架构**：
```
自定义采集脚本（Python）
    ↓ 采集数据
保存到数据库/文件
    ↓ 触发
Dify 工作流（分析 + 推送）
```

**实现步骤**：
1. 用 Python 写采集脚本（Playwright、PDF 解析等）
2. 采集结果保存到 SQLite 或 JSON
3. Dify 工作流读取数据
4. Dify 完成分析、分类、推送
5. Dify 知识库保存历史

**优点**：兼顾灵活性和开发效率
**缺点**：需要写部分 Python 代码

## 三、Coze（扣子）方案详解

### 3.1 Coze 是什么？

Coze 是字节跳动推出的 AI Bot 开发平台，支持：
- 可视化 Bot 构建
- 内置多种插件（搜索、天气、新闻等）
- 支持自定义插件（API 调用）
- 支持知识库
- 支持多平台发布（飞书、微信、网页）

**官网**：https://www.coze.cn/

### 3.2 Coze 实现情报系统的方案

#### 架构设计

```
┌─────────────────────────────────────────────────────┐
│              Coze Bot（情报分析助手）                 │
├─────────────────────────────────────────────────────┤
│  1. 自定义插件 → 调用采集 API                        │
│  2. LLM 对话 → AI 分析                               │
│  3. 知识库 → 历史情报检索                            │
│  4. 工作流 → 自动推送                                │
└─────────────────────────────────────────────────────┘
                        ↓
┌─────────────────────────────────────────────────────┐
│              发布到飞书/钉钉/网页                     │
└─────────────────────────────────────────────────────┘
```

#### 优势

✅ **对话式交互**：用户可以问"最近有哪些国密政策？"
✅ **多平台发布**：直接发布到飞书、钉钉
✅ **插件生态**：可以调用现成插件
✅ **知识库强大**：自动向量化，支持 RAG

#### 劣势

❌ **定时任务弱**：主要是对话式，不适合自动化
❌ **采集能力有限**：需要自己开发插件
❌ **云端为主**：数据安全性相对较低
❌ **扩展性一般**：受限于平台能力

### 3.3 Coze 实施方案

#### 方案：Coze Bot + 自定义插件

**实现步骤**：
1. 开发采集 API（FastAPI）
2. 在 Coze 创建自定义插件，调用采集 API
3. 创建 Coze Bot，配置插件和知识库
4. 用户通过对话触发采集和分析
5. Bot 自动推送到飞书/钉钉

**优点**：对话式交互，用户体验好
**缺点**：不适合完全自动化，需要人工触发

### 4.1 根据需求选择

**适合**：

## 八、Dify 快速上手指南

```bash
# 克隆仓库
git clone https://github.com/langgenius/dify.git
cd dify/docker

# 启动服务
docker-compose up -d

# 访问
# http://localhost:3000
```

### 8.2 创建第一个工作流

#### 步骤 1：创建应用

1. 登录 Dify
2. 点击"创建应用"
3. 选择"工作流"类型

#### 步骤 2：配置节点

**节点 1：HTTP 请求（采集）**
```yaml
方法: GET
URL: https://www.nca.gov.cn
输出变量: raw_html
```

**节点 2：LLM（分析）**
```yaml
模型: deepseek-chat
Prompt: |
  分析以下网页内容，提取最新政策信息：
  {{raw_html}}
  
  输出 JSON 格式：
  {
    "title": "政策标题",
    "date": "发布日期",
    "summary": "摘要",
    "impact": "影响分析",
    "priority": "高/中/低"
  }
输出变量: analysis_result
```

**节点 3：HTTP 请求（推送）**
```yaml
方法: POST
URL: https://oapi.dingtalk.com/robot/send?access_token=xxx
Body: |
  {
    "msgtype": "markdown",
    "markdown": {
      "title": "情报速递",
      "text": "{{analysis_result}}"
    }
  }
```

#### 步骤 3：测试运行

1. 点击"运行"
2. 查看每个节点的输出
3. 调试和优化

### 8.3 配置知识库

1. 创建知识库
2. 上传历史情报文档
3. 在工作流中添加"知识库检索"节点

## 十一、总结

### 核心观点

1. **Dify 和 Coze 完全可以用**，而且能大幅降低开发门槛
2. **混合方案最佳**：采集用 Python，分析用 Dify
3. **快速验证优先**：先用 Dify 验证，再决定是否深度开发
4. **不要过度设计**：从简单开始，逐步迭代

### 关键建议

- ✅ 测试工程师首选 Dify + Python 混合方案
- ✅ 快速验证用纯 Dify
- ✅ 对话交互用 Coze
- ✅ 完全自主用 LangGraph

### 风险提示
